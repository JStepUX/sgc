# 2026-06

## 2026-06-07 — Retire confidence grading; repurpose the per-turn channel into `<turn-summary>`

The constitutional-memory tier was **graded**: each memory carried a 0–100
confidence the model re-scored every turn via a trailing
`<turn-meta>{ "confidence_scores": {…} }</turn-meta>` block (bar + sparkline +
"Confidence Deltas" card). That grading is **gone**. The same per-turn metadata
channel is repurposed into a structured **turn summary**. This touches only the
tier where a model is *already* in the loop — the cosine grep, the local buffer,
and the "no model in retrieval" / "one API call per turn" invariants are
untouched.

- **Memories become plain facts.** `Memory` is now `{ id, text }` — no
  `confidence`, no `history`. The user still curates them (add/edit/delete); the
  model no longer scores them. `MemoryPanel` drops the `%`, the bar, and the
  sparkline; the prompt renders `[M1] <text>` with no confidence tag; the
  `CONFIDENCE SCORING` prompt section and the `DEFAULT_PERSONA` "scored by
  confidence" phrasing are removed.
- **`<turn-summary>` channel.** Sal now ends each turn with
  `<turn-summary>{ "persistent": [...], "volatile": [...], "established_patterns": [...] }</turn-summary>`.
  `persistent` = true until explicitly changed; `volatile` = shifted this turn;
  `established_patterns` = behavioral rules demonstrated. The hardened delimiter
  machinery is reused as-is — `parseTurnResponse` (last-tag anchoring, trailing-
  block validation) and `stripStreamingMeta` (split-chunk hide) only change the
  tag string + validated schema. Missing/malformed lists coerce to `[]`; a
  trailing JSON block with none of the three keys is left as prose.
- **Runs fresh, never fed back.** The summary is a per-turn *observation*, not
  accumulated memory — identical lifecycle to the old `<turn-meta>` (snapshot
  only, no carry-forward into the next prompt). `TurnSummary` lives in
  `lib/types.ts`.
- **Render.** In-message: one **dimmed, always-on, flattened** line beneath the
  reply ("a, b, and c" — all three sections concatenated, section labels
  dropped, empty → nothing), so it respects the reading column's vertical space.
  The structured view (labelled `persistent`/`volatile`/`patterns` lists) lives
  in the inspector's new **Turn Summary** card — the diagnostics surface — which
  replaces the vacated "Confidence Deltas" card. The summary persists in the
  turn's `inspector_json` blob and rehydrates onto the message on reload.
- **DB migration.** `MEMORIES_SCHEMA_SQL` drops the `confidence` column and the
  `memory_history` table. A one-time destructive migration (mirroring the
  existing global→per-chat drop) rebuilds `memories` clean when a legacy
  `confidence` column is detected — the scores + sparkline history are discarded
  by design (same rationale as the prior drop: local research-prototype store).
  `getMemories`/`saveMemories`, the `PUT /api/memories` validator, and the client
  persistence layer (`reassembleMemories` removed) all narrow to `{ id, text }`.
- **Tests.** `prompt.test.ts` rewritten for the new tag/schema (parse, coercion,
  prose-mention safety, split-chunk hide, the persona-tail invariant now pins
  `TURN SUMMARY` / `persistent` / `volatile` / `established_patterns`).
- Verified end-to-end (Playwright, live model): memory card shows no `%`; a turn
  emits the flattened dimmed line beneath the reply and the structured inspector
  card; no "Confidence Deltas"; "1 API call this turn" intact. `typecheck`,
  `lint`, `test`, and `build` all green.
- **Review hardening.** Two follow-ups from a peer review:
  - *Parser robust to the tag appearing inside a summary string.* `confidence_scores`
    held only numbers, so `parseTurnResponse`'s `lastIndexOf(<turn-summary>)` /
    `indexOf(</turn-summary>)` anchors were safe. The free-form summary strings
    aren't: a value like `"asked about <turn-summary> tags"` (or a `</turn-summary>`
    mention) would anchor *inside* the JSON, fail the parse, and leave the raw
    block visible in the finalized message. The parser now anchors the CLOSE on
    the LAST `</turn-summary>` and finds the OPEN by scanning candidates
    front-to-back for the first that parses to a summary — robust to the literal
    tag on both sides. Two regression tests cover the opening- and closing-tag
    cases. (`stripStreamingMeta` was already safe — it hides from the first
    opener followed by `{`/whitespace, so an inner literal is already in the
    hidden region.)
  - *README de-scored.* `README.md` still described 0–100 confidence scoring (the
    tier table + "Sal emits updated confidence scores"); updated to match the
    plain-fact memories and the `<turn-summary>` channel.

## 2026-06-02 — buildPrompt's task tail stays persona-neutral

`buildPrompt`'s **always-appended tail** (`prompt.ts` — the persona-independent
block carrying `YOUR TASK` / `CONFIDENCE SCORING` / the `<turn-meta>` contract)
is meant to be pure *mechanism*; personality belongs to the persona. Two changes
keep it that way. Both are prompt text only — no model call, no memory-path change.

- **Diagram capability added (it was missing).** Native mermaid rendering (entry
  below) was reactive — Sal had no idea it could produce diagrams, so in practice
  it never did. One sentence now states the capability: *"When a diagram would
  clarify structure or flow, emit a mermaid fenced code block (default flowchart
  TD) — it renders natively for the person."* Placed in the tail, **not**
  `DEFAULT_PERSONA`, so it survives a custom-persona swap — a custom persona can
  no more drop diagram support than it can drop confidence scoring. `flowchart
  TD` added to the `TAIL_MARKERS` list in `prompt.test.ts`, so both persona-tail
  tests fail if the line is reverted. (`TD` is already mermaid's default; the
  clause mainly pins it so Sal doesn't drift to `LR`.)
- **Demeanor removed (it didn't belong).** Task step 1 read "Respond to the
  user's input *naturally and helpfully*, informed by…". Tone is the persona's
  job, and the tail appends *after* the persona — so those adverbs silently
  overrode any persona wanting a different demeanor (a deliberately terse or
  adversarial one couldn't take hold). Now just "Respond to the user's input,
  informed by…"; the persona owns voice end to end.

## 2026-06-02 — Native Mermaid diagram rendering in Sal's replies

Fenced `mermaid` blocks in an assistant turn now render as diagrams instead of
showing raw source. The markdown pipeline (react-markdown + remark-gfm +
rehype-quotes) is otherwise unchanged; this hooks the existing `code` / `pre`
component overrides in `AssistantMessage` (`SalienceGatedCognition.tsx`).

- **New component** `src/client/components/MermaidBlock.tsx`. Lazy-loads
  `mermaid@11` via a dynamic `import()` — the build splits it into its own chunks
  (`mermaid.core` ~619 KB + per-diagram-type chunks), so none of it lands in the
  main bundle; it downloads only the first time a diagram appears. mermaid is
  initialised once with `securityLevel: 'strict'` (output sanitised at source via
  its bundled DOMPurify) and `theme: 'base'` whose `themeVariables` map to the Sal
  design tokens (warm near-black field, `fg-1` text, `fg-3` edges).
- **Render only when finalized.** The `code` override gates on `!streaming`: a
  mid-stream block is incomplete and mermaid would throw, so it renders as a
  normal code block until the turn closes, then swaps to the diagram. The `pre`
  override strips its code-box chrome for a finalized mermaid child so the diagram
  isn't double-framed.
- **Graceful failure.** Invalid/unsupported diagram syntax falls back to the same
  styled code block the rest of the chat uses — a bad diagram never crashes a
  message.
- Presentation-layer only: no model in the memory/retrieval path, no extra API
  call, Sal stays ephemeral. Phase 1.5 invariants untouched.
- Verified end-to-end: seeded a sample flowchart into a throwaway chat via the
  model-free `manual-turns` endpoint, confirmed it renders themed on the dark
  field in the live app with zero console errors (Playwright), then deleted the
  test chat.

## 2026-06-01 — Constitutional memories go per-chat (and stop shipping defaults)

Constitutional memories were **global** — one shared set across every chat — and
the app seeded three hardcoded defaults so a fresh install "had something to push
on." Both are now gone: memories are **scoped per chat** and **start empty**.

- **Data model.** `memories` gains a `chat_id` FK (`ON DELETE CASCADE`), so each
  chat owns its own constitutional set and deleting a chat drops its memories
  with it. The schema DDL is hoisted into a `MEMORIES_SCHEMA_SQL` const shared by
  the fresh-create path and a one-time migration: a legacy table (no `chat_id`)
  is dropped and rebuilt — the old global rows belong to no chat, so they're
  discarded by design (confirmed acceptable for the local research-prototype DB).
  Every memory statement (`getMemories`, `saveMemories`, the reconciliation
  id-sweep, the upsert) is now scoped to `chat_id`; critically the save's
  delete-absent sweep only touches *this* chat's rows, never another chat's.
- **Transport.** A chat's memories ride along in its `loadChat`
  (`GET /api/chats/:id`) payload now — the standalone `GET /api/memories` route
  and the client `getMemories()` are removed (one load hydrates turns + persona +
  memories together). `PUT /api/memories` carries a required `chatId`.
- **Client.** `DEFAULT_MEMORIES` is deleted; `memories` initialises to `[]`.
  Hydration and chat-switch load the active chat's set from the detail and reset
  the dirty ref so the programmatic load isn't mistaken for an edit; "Begin again"
  empties the set; the debounced save now keys on the active `chatId`. Net effect:
  add memories in chat A, switch to B → B is empty, switch back → A's are intact
  and distinct.
- **Prompt.** With an empty set the `CONSTITUTIONAL MEMORIES:` block renders
  `(none yet …)` instead of a blank section under the "you carry constitutional
  memories" framing — so the prompt never claims memories that don't exist. New
  pinned test covers the empty case; the non-empty path is byte-identical.
- Still inside the Phase 1.5 invariants: pure curation, no model in the memory
  path, no new API call — only the *scope* and *seed* of the tier changed.
- **Review hardening.** Four follow-ups from a critical review:
  - *No lost edits on chat swap.* The 250ms save debounce was cancelled when a
    chat switch / "Begin again" replaced the outgoing set within the window —
    silently dropping the edit (a regression from the per-chat swap). The swap
    callbacks now `flushPendingMemorySave()` first (reads live refs), and the
    debounce clears the dirty flag as it fires so `dirty` means "unsaved
    changes," keeping the flush a no-op when nothing is pending.
  - *Cross-chat id guard.* `saveMemories` now rejects (`409`) an incoming memory
    id already owned by a different chat. Can't happen via the UI (UUIDs, never
    copied), but the id-keyed upsert would otherwise silently rebind the owner's
    row — defense made loud rather than a `(chat_id, id)` schema rework.
  - *Missing-chat saves are `404`, not `500`.* The PUT now surfaces "chat not
    found" (a check in `saveMemories`, mirroring `saveTurnPair`), which also
    fixes an empty payload to a deleted chat silently returning `200`.
  - Stale "memories are global" comment on `startNewChat` corrected.

## 2026-06-01

- **Brain surgery — manual timeless memories.** The chat memory editor can now
  *insert* turns, not just gate them. An "Add memory" button (top-right, beside
  the close X) opens a form capturing a full user+assistant turn; on submit the
  pair is inserted as the **oldest** turn in the chat and flagged `timeless`.
  Timeless turns are greppable like any other (still pure TF-IDF, no model) but
  the time scorer **negates recency** for them — `timeScore` is pinned to 1.0
  regardless of age or any time phrase in the query, so a curated fact ranks on
  concept alone and never decays. The cosine cliff still holds (concept 0 → no
  retrieval), so a timeless flag can't surface an off-topic memory. In the
  prompt these read `[Turn N · timeless]` instead of a relative-time tag.
  - **Data model.** New `turns.timeless` column (additive migration, default 0).
    `prependManualTurnPair` inserts below the minimum ordinal (negatives are
    fine — ordinals only sort) and does NOT bump `updated_at` (a manual memory
    is the *oldest* content, not new activity, so the chat shouldn't jump the
    history list). `deleteManualTurnPair` removes both halves of a pair to keep
    the user/assistant alternation the cosine engine relies on, and refuses any
    non-timeless turn. New routes: `POST /api/chats/:id/manual-turns`,
    `DELETE /api/chats/:id/turns/:turnId`.
  - **UI.** Timeless cards render with an ember gradient fill, a `timeless` tag,
    and a delete (trash) control in place of the on/off toggle; they're excluded
    from gating and mass actions (always retrievable).
- **Invariant enforced at the mutation, not just the UI.** `setTurnsActive`'s
  UPDATE now carries `AND timeless = 0`, so gating a timeless turn off is a
  silent no-op even if an API caller or future code path hands it a timeless id
  — the "manual memories are always retrievable" promise no longer relies on the
  UI omitting a toggle. (Core value: build the check, don't trust the self-report.)
- **Fixes from review.** Adding/deleting a manual memory now also refreshes the
  chat summaries (`handleTurnsMutated`) — without it, inserting the first turn
  into a previously-empty chat left a stale title/snippet in the history list.
  The Add Memory form now contains keyboard focus to its own sheet (a form-local
  trap) so Tab can't reach the behind-overlay chrome, and the dialog focus-trap
  selector now includes `textarea`/`select`.
